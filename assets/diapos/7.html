Network Policies & Microsegmentación

Implementación de seguridad Zero Trust mediante políticas de red Kubernetes (Namespace: dev)

Seguridad Base
default-deny-ingress Block All

Política fundamental que bloquea todo el tráfico entrante por defecto en el namespace. Ningún pod puede recibir tráfico a menos que sea explícitamente permitido.

api-gateway-allow-external Ingress

Única excepción para tráfico externo. Permite conexiones desde el Ingress Controller / LoadBalancer hacia el API Gateway en puerto 8080.

postgres-allow-backend Database

Restringe el acceso a la base de datos (5432) exclusivamente a los microservicios de backend identificados. Bloquea acceso directo no autorizado.

Acceso a Servicios
allow-from-api-gateway Routing

Permite que el API Gateway se comunique con todos los microservicios downstream (user, product, order, etc.) para enrutamiento de peticiones.

[service]-allow-gateways Specific

6 políticas individuales (una por microservicio: user, product, order, payment, shipping, favourite) que definen granularmente quién puede acceder a cada API específica.

proxy-client-allow-external External

Permite tráfico externo hacia el proxy client, habilitando comunicación desde fuera del cluster hacia este componente de enrutamiento.

Infraestructura
eureka-allow-all-services Discovery

Habilita puerto 8761 para que todos los pods puedan registrarse y enviar heartbeats al Eureka Server para Service Discovery dinámico.

cloud-config-allow-all Config

Permite acceso al Config Server (9296) durante el arranque de los servicios para obtener configuraciones centralizadas y secretos.

allow-prometheus-scraping Metrics

Permite que el servidor Prometheus (desde namespace monitoring) acceda a los endpoints /actuator/prometheus de cada pod.

zipkin-allow-all Tracing

Habilita el envío de spans de tracing (puerto 9411) desde cualquier microservicio hacia el colector Zipkin para observabilidad distribuida.
